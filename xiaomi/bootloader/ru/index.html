1. Архитектура бинарника

Бинарник LK представляет собой сырой бинарник со специфическим заголовком Mediatek:

Offset | Size | Description --------|------|------------ 0x000 | 4 | Magic/ID1: 0x58881688 0x004 | 4 | Image size: 0x001BF668 0x008 | 2 | "lk" magic 0x00A | 2 | Padding 0x030 | 4 | Magic/ID2: 0x58891689 0x034 | 4 | Code offset: 0x200 0x038 | 4 | Version: 0x1 0x200 | - | ARM32 exception vectors (B handlers) 0x224 | - | Early init: CP15 config, cache setup, relocation 0x280 | - | Stack init for all CPU modes 0x25620+| - | Exception handlers (Undef, SWI, Prefetch, Data, IRQ, FIQ)

Код написан на ARMv7-A Thumb2 (interworking). Начальный запуск в режиме ARM32, переключение на Thumb2 через BX после релокации.

2. Обзор процесса загрузки

Preloader (BootROM) → LK (lk_a) → Boot image (boot_a) → Android

Последовательность запуска LK

  1. Векторы исключений по 0x200 (B к обработчикам)
  2. Ранняя инициализация (ARM32):
      - Чтение управляющего регистра CP15
      - Отключение L1 кэша, MMU
      - Саморелокация: копирование образа из адреса загрузки в link-адрес 0x48200000
      - Переход на перемещённый вход 0x48200080
  3. Инициализация стека: Настройка SP для режимов IRQ, FIQ, SVC, ABT, UND
  4. Инициализация платформы: Тактирование MT6877, PMIC, UART, дисплей
  5. Верификация образа: Проверка boot/recovery через AVB 2.0
  6. Управление состоянием загрузки: Чтение SECCFG, установка verifiedbootstate в cmdline
  7. Загрузка ядра: Загрузка и переход в ядро

3. Команда Fastboot OEM Unlock

Регистрация команды

Команда fastboot oem unlock регистрируется через таблицу команд fastboot со строкой -fastboot_oem_lock (по 0x11c4ff). Ведущий - указывает, что это OEM-специфичная команда.

(0x11c56c) │ ├─ "Or, device will power off in %ds" │ ├─ "If you unlock the bootloader,you will be able to install custom operating" │ ├─ "Yes (Volume UP):Unlock(may void warranty)" (0x11c764) │ └─ "No (Volume Down):Do not unlock bootloader" (0x11c794) │ ├─ Token verification (if token provided) │ └─ mi_gen_unlock_token → miunlock verity → token SN verify │ ├─ Data wipe │ └─ unlock_erase (0x11c514) → factory reset │ └─ "To prevent unauthorized access to your personal data,unlocking the bootloader" │ ├─ Set lock state │ ├─ mi_set_lock_state(STATE_UNLOCKED) │ ├─ seccfg_set_lock_state(STATE_UNLOCKED) │ └─ "set lock failed - Err:0x%x" (0x11ca00) │ ├─ ATF hardware lock │ └─ set_hw_locked_in_atf(0) → SMC call to TrustZone │ └─ Reboot

4. Слой MiCustom

Xiaomi реализует собственный слой разблокировки (отдельно от стандартного AOSP fastboot unlock). Функции MiCustom слинкованы как отдельный модуль в LK.

API Surface

┌─────────────────────────────────────────────────┐
│              MiCustom Interface                  │
├─────────────────────────────────────────────────┤
│ mi_get_lock_state()       → reads lock state    │
│ mi_set_lock_state(state)  → writes lock state   │
│ mi_gen_unlock_token()     → generates token     │
│ miunlock verity()         → verifies token      │
│ miunlock set sigdata()    → stores signature    │
│ mi_check_magic()          → checks RPMB magic   │
│ mi_get_magic()            → reads RPMB magic    │
│ mi_set_magic()            → writes RPMB magic   │
│ mi_get_sig()              → reads signature     │
│ mi_set_sig()              → writes signature    │
│ Custom_error_process()    → error handler       │
└─────────────────────────────────────────────────┘

Детали функций

mi_get_lock_state()

  • Смещение в файле: строки по 0xa8130
  • Вызывает MiCustom Get LockState → чтение из SECCFG
  • Возвращает текущее состояние блокировки (0=locked, 1=unlockable, 2=unlocked)

mi_set_lock_state(state)

  • Смещение в файле: строки по 0xa811c
  • Вызывает MiCustom Set LockState → запись в SECCFG
  • Проверяет переход состояния: "%s want to set wrong lock state : %d" (0xa8488)
  • При ошибке: Custom_error_process с отчётом seccfg_state

mi_gen_unlock_token()

  • Смещение в файле: строки по 0xa86c8
  • Генерирует токен разблокировки, привязанный к идентификатору устройства
  • Использует:
      - hash(ufsid) — хеш ID устройства UFS (0xa82b0)
      - cpuid — уникальный ID CPU (0xa8300, 0xa8340)
      - RSA дешифрование (rsa_exptmod по 0xa8200)
      - Base64 кодирование для формата токена (0xa8540)
  • Токен содержит подпись над идентификаторами устройства

miunlock verity()

  • Смещение в файле: 0xa8524
  • Проверяет подпись токена разблокировки
  • Проверяет token SN verify success (0xa8388) → верификация серийного номера
  • Валидирует sig_len и данные подписи
  • Возвращает miunlock verity failed, result=0x%x при ошибке
  • Возвращает wrong unlock data (0xa84c4) при неверном формате
  • Возвращает invalid argument for miunlock (0xa84d8) при плохих аргументах

Поток токена MiCustom

fastboot oem unlock <token> │ ├─ miunlock verity() │ ├─ Decode base64 token │ ├─ Extract signature and device SN │ ├─ RSA verify: hash(device_id) == decrypted(signature) │ │ ├─ RSA modulus: stored in binary at 0xa85c0 (272 bytes) │ │ └─ RSA exponent: 0x10001 (65537, standard) │ ├─ "token SN verify success" or "token sig not same" │ └─ Return result │ ├─ miunlock set sigdata() │ └─ Store verified signature in RPMB for audit │ └─ mi_set_lock_state(UNLOCKED)

5. SECCFG

SECCFG (Security Configuration) — постоянное хранилище состояния блокировки, обычно находящееся в разделе seccfg или NVRAM.

Структура хранилища

seccfg partition layout: ├── SECCFG_MAGIC (4 bytes) - validation magic ├── lock_state (uint32) │ 0 = LOCKED (STATE_LOCKED) │ 1 = UNLOCKABLE (STATE_UNLOCKABLE, can be unlocked) │ 2 = UNLOCKED (STATE_UNLOCKED) ├── unlock_ability (uint32) - permission flag ├── dm_verity_state │ └─ seccfg_get_dm_verity_state / seccfg_set_dm_verity_state └── backup copy (redundant storage)

Функции

seccfg initialization: ├─ "Initializing seccfg major" (0x1371e8) ├─ "Initializing seccfg bak" (0x13720c) ├─ "Initializing seccfg major NV" (0x137074) ├─ "Initializing seccfg bak NV" (0x137098) ├─ "Show seccfg status" (0x1370bc) ├─ "seccfg not found!" (0x13722c) └─ "fatal error!" (0x137250) Seccfg validation: ├─ SECCFG_MAGIC_CORRECT / SECCFG_MAGIC_INCORRECT ├─ SECCFG_VALID / SECCFG_NOT_VALID └─ Both major and backup copies validated Seccfg operations: ├─ seccfg_get_lock_state() - read lock state ├─ seccfg_set_lock_state(state) - write lock state ├─ seccfg_get_dm_verity_state() └─ seccfg_set_dm_verity_state()

Обработка ошибок

Custom_error_process(err, cust_state, seccfg_state): Format: "%s: err: 0x%x, cust_state: %d, seccfg_state: %d" ├─ "Seccfg set lock state failed!" (0xa7e64) ├─ "Custom set lock state failed!" (0xa7e84) └─ Params: error code, MiCustom state, seccfg state

6. Взаимодействие с ATF TrustZone

Загрузчик взаимодействует с ARM Trusted Firmware (ATF) через инструкции SMC (Secure Monitor Call).

Протокол SMC

  • Тип вызова: Fast SMC (не Yielding)
  • Используемый immediate SMC: 0xF0 (наиболее частый), другие: 0xD0, 0xC0, 0xA0, 0xB0
  • FID, хранящиеся в пулах литералов:
      - 0xC2000000 — MTK custom fast SMC
      - 0x82000000 — Standard fast SMC (OEM/service)
  • Строки перехода:
      - "[Sec] We are going to enter ATF for smc call" (0xa7e1f)
      - "[Sec] We are back from ATF" (0x1383f5)

Функция set_hw_locked_in_atf()

  • Строка по: 0x12b290
  • Назначение: Блокирует/разблокирует аппаратный предохранитель через ATF
  • Делает SMC вызов к ATF для установки состояния аппаратной блокировки
  • Вызывается как при блокировке, так и при разблокировке
  • Параметры SMC вызова:
      - R0 = SMC function ID (0xC20000xx или 0x820000xx)
      - R1 = устанавливаемое состояние блокировки (0 = unlock, 1 = lock)
      - R2+ = дополнительные параметры

Расположение SMC вызовов ATF

Ключевые SMC вызовы разбросаны по всему коду:

  • Ранняя загрузка: 0x15a8, 0x1b7c, 0x1c00+ (серия SMC при загрузке)
  • Инициализация платформы: 0x33b8, 0x3a24, 0x3d96 (DEVAPC, управление питанием)
  • SPM (System Power Management): 0x6c34+ (SPM SMC вызовы)
  • Операции SECCFG: 0x1bed88+ (запись SECCFG через SMC)
  • Операция разблокировки: область 0x12403c (рядом с функциями разблокировки SECCFG)

Идентификаторы функций SMC ATF:

FIDРасположениеНазначение
0xC20000000x1c1dacБаза MTK custom SMC
0x820000000x1c44c0+Fast SMC (OEM/service)
Sub-FIDsРазличныеLock/unlock, SEJ, RPMB

Аппаратные предохранители блокировки

Безопасный мир ATF управляет аппаратными предохранителями (efuse) для:

  • SBC (Secure Boot Control): Принудительная аппаратная безопасная загрузка
  • SDL (Security Debug Lock): Контроль доступа к отладочному порту
  • Читаются через: dump sw sbc:0x%x, sw sdl:0x%x , hw sbc: %d (0x136fb8)

7. RPMB

RPMB (Replay Protected Memory Block) используется для:

  1. Программирования ключа аутентификации — хранит общий секрет
  2. Конфигурации защиты записи — предотвращает вмешательство в разделы
  3. Журнала аудита разблокировки — хранит доказательства подписи разблокировки

Программирование ключа RPMB

┌──────────────────────────────────────────────┐ │ RPMB Authentication Key Programming │ ├──────────────────────────────────────────────┤ │ "1st time run RPMB Lock, write magic" │ │ rpmb_authen_read_counter() │ │ rpmb_authen_secure_write_protect_conf_block_ │ │ check() / read() / write() │ │ ufshcd_authen_data_write_rsp_check() │ │ rpmb_get_mac() │ ├──────────────────────────────────────────────┤ │ Results: │ │ "[PASS]Authentication Key Program Success" │ │ "[Failed]Authentication Key not Programed" │ │ "Authentication failure(MAC not match)" │ │ "Authentication key not yet programmed" │ └──────────────────────────────────────────────┘

Хранение токена разблокировки в RPMB

Слой MiCustom хранит данные, связанные с разблокировкой, в RPMB:

  • mi_set_magic() / mi_check_magic() — магическое значение RPMB для состояния блокировки
  • mi_set_sig() / mi_get_sig() — подписи разблокировки, хранящиеся в RPMB
  • rpmb_read() / rpmb_write() по 0xa81b0 и 0xa81c0

Защита записи

Дополнительные компоненты:

  • rsa_exptmod — функция модульного возведения в степень RSA (0xa8200)
  • RSA_padding_add — проверка padding PKCS
  • Параметр tmp_key для операций RSA

OEM открытый ключ

g_sec_oemkey (по смещению в файле 0x137bdc) — глобальная переменная, хранящая OEM открытый ключ для верификации подписанных образов.

HMAC

HMAC используется для аутентификации RPMB:

  • "rpmb_hmac error" — ошибка верификации HMAC
  • "Authentication failure(MAC not match)" — проверка MAC не пройдена
  • "compare hmac error" — несовпадение при сравнении

Структура токена

Unlock token format: ├─ Base64 encoded (charset: "ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/") ├─ Contains: device SN + RSA signature ├─ Size: sig_len variable ("no sig stored", "sig_len = %d") └─ Verified against: hash(ufsid) + cpuid + serial number

Хеш

SHA-256 используется для различных операций:

  • "ERROR: CAN NOT SHA256(UFSID)" — хеширование ID устройства UFS

10. Полная последовательность разблокировки

Фаза 1: Приём команды
fastboot oem unlock [data|token] ├─ fastboot parses command ├─ Matches "-fastboot_oem_lock" command handler └─ Enters unlock handler with [data] parameter
Фаза 2: Проверка разрешения
├─ Check unlock_ability from SECCFG │ ├─ Read unlock_allowed_flag_offset │ └─ If unlock_ability == 0 → "Failed to get unlock permission" → DENY ├─ Read device lock state │ ├─ mi_get_lock_state() │ │ └─ MiCustom Get LockState → seccfg_get_lock_state() │ └─ If already unlocked → no action needed └─ If locked with unlock ability → continue
Фаза 3: Подтверждение пользователя (без токена)
├─ Display: "Unlock bootloader?" ├─ Warning: data will be wiped ├─ Wait for volume key input │ ├─ Volume UP → continue unlock │ └─ Volume Down / timeout → cancel └─ Power-off countdown on timeout
Фаза 4: Верификация токена
├─ miunlock verity(unlock_data) │ ├─ Validate input: "wrong unlock data" │ ├─ Decode base64 token │ ├─ Extract serial number + signature │ ├─ Compute hash(ufsid + cpuid + serial) │ ├─ RSA verify: signature == hash │ │ ├─ rsa_exptmod(public_key, signature) │ │ ├─ padding_unpad(result) │ │ └─ compare(hash, result) │ ├─ "token SN verify success" → PASS │ └─ "token sig not same" → FAIL → "Unlock failed - Err:0x%x" └─ On verify failure → abort
Фаза 5: Стирание данных
├─ unlock_erase() │ ├─ Wipe userdata partition │ └─ "factory data reset" ├─ Display: "To prevent unauthorized access to your personal data,unlocking the bootloader" └─ Clear personal data
Фаза 6: Смена состояния
├─ mi_set_lock_state(STATE_UNLOCKED) │ ├─ Validate state transition │ ├─ seccfg_set_lock_state(UNLOCKED) │ │ ├─ Read SECCFG block │ │ ├─ Validate magic + checksum │ │ ├─ Write unlock state │ │ ├─ Write backup copy │ │ └─ Verify write ("Major SECCFG_VALID", "Bak SECCFG_VALID") │ └─ Error: "Seccfg set lock state failed!" / "Custom set lock state failed!" ├─ miunlock set sigdata(token_signature) │ └─ Store unlock token in RPMB audit trail └─ "Set_RTC_Fastboot_Mode" → RTC flag for next boot
Фаза 7: Аппаратное состояние ATF
├─ set_hw_locked_in_atf(0) // 0 = unlocked │ ├─ Prepare SMC call to TrustZone │ ├─ "[Sec] We are going to enter ATF for smc call" │ ├─ SMC #x → ATF secure monitor │ │ ├─ FID: 0xC20000xx / 0x820000xx │ │ ├─ Args: unlock_state, flags │ │ └─ ATF writes hardware fuse/register │ └─ "[Sec] We are back from ATF" └─ Hardware lock state now reflects unlocked
Фаза 8: Обновление состояния загрузки
├─ Set kernel cmdline: │ └─ androidboot.verifiedbootstate=orange ├─ AVB state: allow_unverified_slots = true ├─ Set lock_state in SECPOLICY │ └─ "[SEC_POLICY] lock_state = UNLOCKED" └─ Display: orange_state_warning at boot logo
Фаза 9: Перезагрузка
├─ Save RTC fastboot flag ├─ Reboot device └─ LK reloads → detects unlocked state ├─ Skips AVB verification ├─ Sets ORANGE boot state └─ Allows custom images

Значения состояний блокировки

SECCFG lock_state: 0 = LOCKED - Full AVB verification enforced - verifiedbootstate=green - Only signed official images boot - fastboot oem unlock → "Unlock bootloader?" 1 = UNLOCKABLE (bootloader allows unlock) - Can transition to UNLOCKED with valid token - Verified by unlock_ability flag in SECCFG 2 = UNLOCKED - AVB verification disabled - verifiedbootstate=orange - Warning displayed at boot - Custom images allowed - fastboot oem lock → "Lock bootloader?" 3 = LOCKED_CRITICAL (if implemented by OEM)
