2. Обзор цепочки загрузки

Стандартная 4-этапная цепочка MediaTek:

Цепочка загрузки
Сброс при включении питания
ЭТАП 1: BROM (Boot ROM) — 128 КБ, в кристалле SoC
Роль: HW инициализация, определение источника загрузки, загрузка preloader + проверка RSA
Вход: ARM @ 0x0 → Thumb @ 0xDA70
ЭТАП 2: Preloader — 472 КБ, загрузка @ 0x100000

Роль: Android OS

3. ЭТАП 1: BROM

Вектор сброса и ранняя инициализация

  • ARM режим при включении питания, BROM отображён в памяти по адресу 0x0
  • Вектор сброса (0x00): B 0x54 — переход через векторы исключений
  • Векторы исключений (0x04-0x3C): все бесконечные циклы (не используются в BROM)
  • Сохранение контекста (0x10-0x3C): сохраняет r0-r4, LR, CP15 c5/c6 конфигурацию MPU в 0x00102B60
  • r0 = выбор устройства загрузки (eMMC/UFS/SDMMC)
  • r1-r4 = конфигурация загрузки
  • Main init (0x54): zero regs, early_hardware_init(), SVC stack @ 0x00013084, secure_init(), clock_init(), watchdog_disable()
  • BLX 0xDA70 → переключение в Thumb режим

Основной BROM (Thumb, 0xDA70+)

  1. Определение источника загрузки: eMMC/UFS/COMBO (строки @ 0x12B4)
  2. Инициализация контроллера хранилища
  3. Проверка заголовка TIMESTMP
  4. Загрузка Preloader: сканирование заголовка GFH (MMM\x01), разбор GFH_FILE_INFO, копирование в SRAM 0x100000, проверка подписи через встроенный открытый ключ
  5. Режим загрузки по USB (USBDL): ACM USB виртуальный COM-порт, ожидает подписанный DA
  6. Переход к preloader: BX @ 0x10066C

BROM НЕ выполняет: инициализацию DRAM, разбор файловых систем, верификацию загрузочного образа.

Паттерн ссылок на строки (LDR+ADD)

LK uses PC-relative LDR + ADD to reference strings (no GOT indirection):

ldr rN, [pc, #imm12]     ; load 32-bit literal from pool
add rN, pc               ; rN = literal + PC → absolute string address

Target computation: lit_val + ((add_addr + 4) &amp; ~3)

Encodings: 16-bit LDR(literal) = 01001 Rt imm8, 32-bit LDR.W = 0xF8DF imm12; 16-bit ADD Rd,PC = 0x4478 | Rd, 32-bit ADD = 0xEB0x prefix.

6. LK Fastboot

Диспетчеризация команд

fastboot_loop:
read USB command into buf
match against registered table (array of {char *name, void (*handler)()})
if matched → call handler(cmd, arg, reply)
if "oem *" → strip "oem ", match subcommand
if not matched → "unknown command"
if not allowed in state → "not allowed in locked state"

Таблица команд в файле 0xAB0C8, цикл диспетчеризации в 0xAAE78. Форматы ответов: OKAY, FAIL<message>, INFO<message>, DATA<size><data>.

Структуры данных

struct fastboot_cmd {
    const char *name;      // e.g. "flash:", "oem unlock"
    void (*handler)(const char *arg, unsigned sz, char *reply);
};

struct seccfg {
    uint32_t magic;         // validation magic

Проверки целостности в LK

LK has multiple integrity checks that may detect and halt on patched code:

  • sha256hw_init/process/done functions at 0xA37E0, 0x137C4B
  • crc32 error! strings at 0x117664, 0x11776A
  • Image hash verification code at 0x12A640
  • ubk_hash used to verify LK's own public key root-of-trust

What likely kills a patched LK (3 hypotheses):

  1. UBK hash mismatch — LK computes hash of loaded key material, compares to fuse; patching LK changes loaded code regions, key-load function behaves differently
  2. Self-CRC32 on code section — preloader-style integrity hash embedded in LK binary; patching a single byte breaks it
  3. Xiaomi custom signature on ATF wrapper — preloader checks ATF header signature before jumping to LK; patching LK invalidates this

  • efuse.bin: ALL 0xAA (unprogrammed erase state), otp.bin: ВСЕ 0x00

Рекомендуемый путь атаки

  1. USB BROM mode (vol up + power)
  2. Kamakiri (Vuln 2) → BROM code exec → dump BROM, verify blacklist addrs
  3. GCPU bypass (Vuln 1) → disable blacklist → full memory access

  6. Write patched preloader to boot partition
  7. Device boots ANY unsigned boot image

Примечание о холодном хранении

Efuse state (all 0xAA) is the single most critical finding:

  • Device never had hardware secure boot enabled — OEM shipped with bootloader permanently unlockable
  • Even a full software restore to stock cannot enable hardware secure boot (efuses were never blown at the factory)
  • Any exploit used once will survive future software updates
  • OTA updates rewrite flash with locked state, but efuses remain unblown → software locks can always be bypassed again via same exploits

Дополнительные заметки по оценке уязвимостей

The preloader exposes several exploitable attack surfaces:

  • CVE-2020-0069 pattern: DRAM initialization via USB download without proper signature check on downloaded images
  • Factory test modes: BR FACTORY BOOT, BR ATE FACTORY BOOT, MT65XX_FACTORY_KEY, MT65XX_RECOVERY_KEY keys detectable

  • No console= restriction in boot cmdline
  • SELinux enforcing (permissive=0) for runtime system only — does not protect boot chain
  • Secure boot chain: Preloader verifies ATF (BL31) via cert chain vfy fail; ATF verifies TEE (BL32) via TEE verify fail with rollback; but vbmeta algorithm=NONE breaks upper chain
