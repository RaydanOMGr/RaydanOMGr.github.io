ЭТАП 1: BROM (Boot ROM)

Вектор сброса и ранняя инициализация

При включении CPU ARM исполняет код по адресу 0x0 в режиме ARM. BROM отображается в память по адресу 0x0.

Вектор сброса (0x00): B 0x54 — прыжок через таблицу векторов исключений.

Векторы исключений (0x04-0x3C): Все бесконечные циклы (undefined, SWI, prefetch abort, data abort — не используются в BROM).

Сохранение контекста (0x10-0x3C):

0x10: LDR  r9, =0x00102B60    ; База области сохранения контекста
0x14: STR  r0, [r9]           ; Сохранить причину загрузки
0x18: STR  r1, [r9, #4]       ; Конфигурация загрузки
0x2C: MRC  p15, 0, r0, c5, c0, {0}  ; Конфигурация региона MPU
0x30: MRC  p15, 0, r1, c6, c0, {0}  ; База региона MPU
0x3C: B    0xD9A8             ; Прыжок в основной init

Сохранённый контекст: r0 (выбор устройства загрузки: eMMC/UFS/SDMMC), r1–r4 (конфиг загрузки), LR, CP15 c5/c6 (конфиг региона MPU для окна устройства загрузки).

Основной init по адресу 0x54

0x54: MOV  r0, #0             ; Обнуление рабочих регистров (r0–r12, SP, LR)
0xB0: BL   0x11B54            ; early_hardware_init() — тактирование, PLL, сторожевой таймер
0xE0: CPS  #0x13              ; Вход в режим SVC
0xE4: MOV  SP, =0x00013084    ; Указатель стека SVC
0xF0: BL   0x20C              ; secure_init() — включение защищённых функций
0xF4: BL   0x11BB0            ; clock_init() — дерево системных тактов
0xF8: BL   0x7B20             ; watchdog_disable()
0xFC: BLX  0xDA70             ; Переход в режим Thumb → main_brom()

Основной BROM (Thumb режим, 0xDA70+)

  1. Определение источника загрузки — чтение HW регистров eMMC/UFS/COMBO (строки: "EMMC_BOOT" @ 0x12B4, "UFS_BOOT", "COMBO_BOOT")
  2. Инициализация контроллера памяти
  3. Проверка заголовка TIMESTMP — валидация разметки flash
  4. Загрузка Preloader — поиск заголовка GFH (MMM\x01 magic), парсинг GFH_FILE_INFO, копирование в SRAM по 0x100000, верификация подписи через встроенный открытый ключ
  5. USB Download Mode — если нажата комбинация клавиш или загрузка не удалась:
      - Режим "USBDL" (строки по 0x3FB0)
      - Режимы ACM USB виртуального COM-порта: "ACM COMMU", "ACM DATA", "ACM VIRTUALCOM"
      - Ожидание подписанного Download Agent (DA) через USB
  6. Переход в preloader — очистка кэшей, BX на вход preloader по 0x10066C

BROM НЕ выполняет: инициализацию DRAM, разбор файловых систем, верификацию boot image.

ЭТАП 2: Preloader

Загрузка и цепочка заголовков GFH

BROM загружает preloader_ruby.bin из загрузочного раздела через сырое чтение блоков. Файл начинается с заголовков GFH:

СмещениеТипРазмерОписание
0x000GFH_FILE_INFO (тип 0)56 BДескриптор образа
0x038GFH_BL_INFO (тип 1)12 BИнформация/версия загрузчика
0x044GFH_BL_SEC_KEY (тип 7)100 BОткрытый ключ RSA (данные по 0x50)
0x0A8GFH_ATTEST (тип 2)20 BДанные аттестации
0x0BCGFH_BL_LOAD (тип 8)48 BДескрипторы сегментов загрузки
0x0ECGFH_BL_ATTR (тип 18)20 BАтрибуты (флаги отладки?)

Парсинг GFH_FILE_INFO:

magic:       MMM\x01
name:        FILE_INFO
flash_type:  0x0F00          ; eMMC загрузочный раздел
sig_len:     0x20 (32)       ; Длина хеша SHA-256
load_addr:   0x100000
entry:       0x66C           ; Вход = load_addr + 0x66C = 0x10066C

GFH_BL_SEC_KEY содержит встроенный открытый ключ RSA для верификации подписи preloader. При неудачной верификации BROM переходит в режим USB загрузки.

Структура файла

0x000-0x0FF:  Заголовки GFH
0x100-0x26C:  Код запуска ARM
0x300-0x340:  Область данных GFH
0x100-0x57FFF: Код Thumb2
0x58000-0x6F000: строки / rodata
0x72000-0x73634: GOT (544 ссылок на строки через паттерн LDR+ADD на основе GOT)

Точка входа (0x10066C)

  • Установка указателя стека (SRAM ~0x200000+)
  • Инициализация PLL, контроллера DRAM, pinmux
  • Переход в режим Thumb2

Инициализация DRAM

Основная задача: инициализация LPDDR4x через шину PMIC/SPI, калибровка таймингов, включение каналов.

После инициализации DRAM:

0x100000-0x17B000  Код Preloader
0x200000+          Стек/куча/BSS
0x4000000+        Доступная DRAM

Парсинг таблицы разделов

Чтение GPT с устройства загрузки (LBA 1 заголовок, LBA 2–33 записи). Ключевые разделы:

Цепочка вызовов верификации:

partition_loader (0xD750)
  → image_loader (0x3E780)
    → RSA_verify_main @ 0x50CD4 (RSA-2048 + SHA-256)
    → image_auth_check @ 0x50F94 (SHA-256 + сравнение хеша)
    → Дополнительные проверки для TEE (0x512B8-0x51400)
    → Проверка хеша прошивки (0x51118-0x5128A)

Поток RSA_verify_main @ 0x50CD4:

push.w {r4-r7,sb,sl,fp,lr}; sub sp,#0x48
  → BL 0x461E0 (инициализация крипто)
  → BL 0x3294C, 0x461FA (загрузка/парсинг данных)
  → BL 0x50B14 (хеш SHA-256)
  → BL 0x5187C (криптооперация)
  → BL 0x50A58, 0x50AC4 (шаги верификации)
  → BL 0x51B98 (верификация подписи RSA)
  → BL 0x51D48 (сравнение хеша)
  → CMP R0,#0; IT NE; MOVNE R7,R3
  → return r7 (=0 при успехе)

Ветки ошибок: 0x50E36, 0x50E9C, 0x50EAC, 0x50EEE, 0x50F04, 0x50F18 → 0x50F32.

Поток image_auth_check @ 0x50F94:

push {r4-r7,lr}; sub sp,#0x2c
  → BL 0x50B14 (SHA-256)
  → BL 0x51D48 (сравнение хеша)
  → CBNZ R0,0x50FD6 → "[%s] img auth fail(0x%x)"
  → CBZ R0,0x50FE6 → "[%s] %s" "Verify Pass"
  → return r4 (=0 при успехе)

Переход в LK

После верификации: парсинг BR заголовка LK, копирование по 0x48200000, установка аргументов загрузки, очистка кэшей, переход на точку входа LK.

; Обработчики исключений
0x200: B    0x224           ; Вектор сброса

0x208: B    0x2563C         ; SWI

0x210: B    0x25674         ; Data abort

; Реальный вход
0x224: LDR  r6, =0x48200020 ; Хранилище источника загрузки
       STR  r4, [r6]        ; Сохранить источник загрузки от preloader
       MRC  p15, 0, r0, c1, c0, {0}  ; Чтение SCTLR
       BIC  r0, r0, #0xB000 ; Очистка битов I-cache + MMU
       BIC  r0, r0, #5      ; Очистка M (MMU) + C (D-cache)
       ORR  r0, r0, #2      ; Установка A (проверка выравнивания)
       MCR  p15, 0, r0, c1, c0, {0}  ; Запись SCTLR
       ; Саморелокация при неверном адресе загрузки
       ; Копирование 0x48200000-0x483BF668 → 0x48200000
       BX   0x48200080      ; Переход на перемещённый код

Обнуление BSS → lk_main

LDR  r0, =0x48338D60  ; Начало BSS
LDR  r1, =0x48338D60  ; Конец BSS (начало = конец? возможно ошибка)

; Цикл обнуления BSS
BLX  0x4827AEC        ; lk_main() — вход в C

LK Main (0x4827AEC) — C Runtime

  1. Инициализация архитектуры: MMU + таблицы страниц, включение D-cache, обслуживание TLB
  2. Инициализация платформы (MT6877): UART консоль, дерево тактирования, GPIO/pinmux для дисплея/кнопок
  3. Инициализация дисплея: Панель, подсветка, framebuffer (логотип загрузки)
  4. Определение устройства загрузки: Чтение PMIC/регистров для причины загрузки:
      - Обычная загрузка
      - Recovery (громкость вверх + питание)
      - Fastboot (громкость вниз + питание)
      - Meta режим
  5. Загрузка boot image: Поиск GPT → чтение заголовка Android boot image v3/v4 → загрузка kernel (Image.gz-dtb) + ramdisk в DRAM → передача DTB
  6. Загрузка ядра: Настройка ATAGs/device tree → режим AArch64 → переход на точку входа ядра

  • patched — улучшенная версия (обход на уровне вызовов, менее заметный: 12 Б)

Перезапись состояния режима загрузки LK (lk_a_patched.bin)

11 байт изменено по 0x273B4-0x27400:

; Оригинал (0x273B4):
CMP  r4, #4
IT   EQ
STREQ.W  r4, [r8]           ; Запись 4 при r4==4
BEQ  0x273EA
...
; По 0x27400:
MOVS r3, #3
STR.W r3, [r8]              ; Запись 3 в другом пути

; Патч:
MOVS r3, #0
STR.W r3, [r8]              ; Всегда запись 0
B    0x273EA
...
; По 0x27400:
MOVS r3, #0
STR.W r3, [r8]              ; Запись 0 вместо 3

Эффект: Значения 4 и 3, записываемые в [r8], заменены на 0. [r8] вероятно управляет блокировкой режима загрузки или состоянием верифицированной загрузки (green/orange/red). Нейтрализует блокировку/проверку безопасности при выборе режима загрузки.

Поверхность уязвимостей

УЯЗВ 1 GCPU Blacklist Bypass (Аппаратный криптодвижок)

Расположение: BROM — GCPU @ 0x10050000 CRITICAL

Тип: Обход ограничений аппаратной безопасности

BROM реализует чёрный список памяти, предотвращающий чтение чувствительных регионов (BROM, SRAM preloader, регистры безопасности). Управляется через регистры GCPU. Регистры чёрного списка доступны для записи через тот же интерфейс AES-CBC, что используется для криптографии — разделение контроля доступа отсутствует.

Эксплойт: Запись 0 в регистры чёрного списка через настройку AES-CBC → полный R/W доступ к памяти.

Влияние: Полный дамп BROM, дамп preloader из SRAM, R/W ко ВСЕЙ памяти.

УЯЗВ 2 Kamakiri — Переполнение USB Control Transfer

Расположение: BROM — интерфейс USB CDC ACM CRITICAL

Тип: Переполнение буфера USB control transfer + манипуляция регистрами

Три варианта:

  • v1: Запись в scratch-регистр watchdog (0x10007050) → перезапись send_ptr → запуск SEND_CERT (команда 0xE0) → выполнение кода в контексте BROM
  • v2: USB CDC line coding ctrl_transfer — отсутствие проверки длины при копировании дескриптора → переполнение стекового буфера → перезапись соседних данных
  • v3 (KamakiriPL): Переполнение USB CDC line coding на уровне preloader → обход состояния аутентификации; также bypass_2nd_da() для обхода авторизации DA

Влияние: Произвольная нагрузка в контексте BROM, дамп preloader/BROM, обход авторизации загрузки 1-го этапа.

УЯЗВ 3 HeapBait — Переполнение кучи в XML парсере (DA)

Расположение: Download Agent — парсер XML команд CRITICAL

Тип: Переполнение буфера кучи (несоответствие выделенного размера и размера чтения)

При обработке DA команды CMD:SECURITY с allinone_signature:

  1. Парсинг атрибута XML filename → выделение буфера на основе ОБЪЯВЛЕННОГО размера имени файла
  2. Чтение данных файла из USB с использованием РАЗМЕРА ПАКЕТА (0x2000), а не выделенного размера
  3. Выделенный буфер: 0x1400 байт; данные за чанк: 0x2000 байт → переполнение: 0xC00 байт

Эксплойт: 3 этапа: загрузка shellcode как signature → запуск переполнения с filename из 5000 символов → повреждённый заголовок аллокатора вызывает перезапись колбэка DPC → выполнение shellcode с привилегиями DA → патч политик безопасности (hash binding, verify policy, download policy, SBC, SLA).

Значения самопроверки (ARM64 на MT6886+):

shellcode_addr = 0x44C7F100, heap_base = 0x4007F100, dpc_addr = 0x40070028

Проверка: DA пропатчен, если содержит "Dest XML file name buffer overflow".

УЯЗВ 4 Efuse/OTP не запрограммированы

Расположение: Efuses @ 0x11F10000, раздел OTP CRITICAL

Тип: Отсутствие постоянной конфигурации безопасности

efuse.bin: ВСЕ 0xAA (незапрограммированное состояние стирания). otp.bin: ВСЕ 0x00.

Отсутствуют критические efuse: включение secure boot, отключение JTAG, хеш ключа OEM, конфигурация загрузки. Устройство не имеет аппаратного корня доверия — только программная безопасность.

Влияние: Любой программный обход является ПОСТОЯННЫМ. Невозможно заблокировать после загрузки. Все efuse R/W через GCPU (Уязв 1).

УЯЗВ 5 Разделы безопасности перезаписываемы

Расположение: Flash разделы seccfg, gsort, sec1, vbmeta_* HIGH

Тип: Программно реализованная безопасность на перезаписываемом носителе

  • seccfg: 64 Б используется из 8 МБ — магия "MMMM", flags=0x01 (включено?), 32-байтный хеш/секрет. Весь раздел можно обнулить.
  • gsort: Минимальные данные с повторяющимися паттернами по 0x70/0x78 (счётчик отката?). Повсюду стёртые паттерны.
  • sec1: Полностью нулевой (2 МБ) — отсутствие состояния блокировки = загрузчик разблокирован.
  • vbmeta: Стандартный заголовок AVB — неактуален при патченом загрузчике.

УЯЗВ 6 Preloader RSA Verify NOP (Подтверждено)

Расположение: preloader_ruby.bin @ 0x50CD4, 0x50F94, 0x3E66C, 0x3E68C HIGH

Тип: Обход верификации подписи RSA (два метода)

Оба метода подтверждены рабочими: патч возврата из функции (return 0) и NOP в месте вызова (пропуск вызова). См. раздел Анализа патчей для подробностей на уровне байт.

УЯЗВ 7 Перезапись состояния режима загрузки LK

Расположение: lk_a.bin @ 0x273B4, 0x27400 MEDIUM

Тип: Перезапись регистра состояния безопасности

Значения 4 и 3, записываемые в [r8], заменены на 0 — нейтрализует заблокированные/принудительные состояния загрузки. См. раздел Анализа патчей для подробностей.

Архитектура безопасности

  1. BROM → Preloader: Проверка подписи RSA-2048 + SHA-256 через ключ в GFH_BL_SEC_KEY. Провал → USB download mode (требуется аутентифицированный DA).
  2. Preloader → LK/boot: Аналогично RSA-2048 + SHA-256. Каждый образ (lk, boot, recovery, tee) верифицируется индивидуально.
  3. LK → Kernel: Android Verified Boot (AVB) — дерево хешей vbmeta + счётчик отката. LK передаёт device tree с состоянием верифицированной загрузки.
  4. Состояние блокировки: Preloader проверяет состояние блокировки загрузчика (раздел sec1). Заблокировано = только подписанные образы. Разблокировано = экран предупреждения + любые образы загружаются.
  5. Авторизация SLA/DAA: Режим загрузки требует SLA (вызов-ответ с ключом OEM) + DAA (подписанный бинарник DA).

Рекомендуемый вектор атаки (наименьшее сопротивление)

1. USB подключение к устройству в режиме BROM (громкость вверх + питание)
2. Kamakiri (Уязв 2) → выполнение кода в BROM → дамп BROM, проверка адресов чёрного списка
3. GCPU bypass (Уязв 1) → отключение чёрного списка → полный доступ к памяти
4. Дамп preloader из SRAM (0x200000-0x240000)
5. Применение патчей preloader (Уязв 6): NOP для вызовов RSA
6. Запись патченого preloader в загрузочный раздел
7. Устройство загружает ЛЮБОЙ неподписанный boot image

Если существует payload HeapBait для MT6877:

1. Отправка подписанного DA → загрузка HeapBait → XML переполнение (Уязв 3)
2. Патч политик безопасности DA: hash binding, verify policy, download policy → все return 0
3. Прямая запись flash → замена LK, boot, recovery неподписанными версиями

Примечание: Cold Storage

Состояние Efuse (все 0xAA) — самый критический вывод:

  • Устройство никогда не имело включённой аппаратной безопасной загрузки
  • Полное программное восстановление до стока не может её включить
  • OEM поставил устройство с постоянно разблокируемым загрузчиком
  • Любой использованный эксплойт переживёт будущие обновления ПО
  • OTA обновления перезаписывают flash с заблокированным состоянием, но efuse не пережжёны → программные блокировки снова обходятся через те же эксплойты
