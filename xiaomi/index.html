2. Boot Chain Overview

Standard Mediatek 4-stage chain:

STAGE 1: BROM (Boot ROM) — 128 KB, in SoC silicon
Role: HW init, boot source detection, preloader loading + RSA verification
Entry: ARM @ 0x0 → Thumb @ 0xDA70
STAGE 2: Preloader — 472 KB, load @ 0x100000
Role: DRAM init (LPDDR4x), GPT parse, LK/boot image loading + RSA-2048 verify

3. STAGE 1: BROM

Reset Vector & Early Init

  • ARM mode at power-on, BROM memory-mapped at 0x0
  • Reset vector (0x00): B 0x54 — jump over exception vectors
  • Exception vectors (0x04-0x3C): all infinite loops (unused in BROM)
  • Context save (0x10-0x3C): saves r0-r4, LR, CP15 c5/c6 MPU config to 0x00102B60
  • r0 = boot device select (eMMC/UFS/SDMMC)
  • r1-r4 = boot configuration
  • Main init (0x54): zero regs, early_hardware_init(), SVC stack @ 0x00013084, secure_init(), clock_init(), watchdog_disable()
  • BLX 0xDA70 → switch to Thumb mode

Main BROM (Thumb, 0xDA70+)

  1. Boot source detection: eMMC/UFS/COMBO (strings @ 0x12B4)
  2. Storage controller init
  3. TIMESTMP header check
  4. Preloader loading: scan GFH header (MMM\x01), parse GFH_FILE_INFO, copy to SRAM 0x100000, verify signature via embedded public key
  5. USB Download Mode (USBDL): ACM USB virtual COM port, waits for signed DA
  6. Jump to preloader: BX @ 0x10066C

BROM does NOT do: DRAM init, filesystem parsing, boot image verification.

String Reference Pattern (LDR+ADD)

LK uses PC-relative LDR + ADD to reference strings (no GOT indirection):

ldr rN, [pc, #imm12]     ; load 32-bit literal from pool
add rN, pc               ; rN = literal + PC → absolute string address

Target computation: lit_val + ((add_addr + 4) &amp; ~3)

Encodings: 16-bit LDR(literal) = 01001 Rt imm8, 32-bit LDR.W = 0xF8DF imm12; 16-bit ADD Rd,PC = 0x4478 | Rd, 32-bit ADD = 0xEB0x prefix.

Deduced Lock State to AVB Mapping

is_device_unlocked = (seccfg.lock_state == 2)
  • lock_state 0 (LOCKED): full AVB enforcement, only signed images
  • lock_state 1 (UNLOCKABLE): fastboot flash allowed, but AVB still rejects custom vbmeta
  • lock_state 2 (UNLOCKED): AVB verification skipped, custom images allowed

read_is_device_unlocked Callback

Strong candidate at 0x7a000 (confirmed in newest analysis):

0x4827a000: push {r4, lr}
  movs r3, #1; str r3, [r1]     ; default: *out_is_unlocked = TRUE
  mov r4, r1
  bl seccfg_parse                 ; returns index
  bl seccfg_get_value(index)     ; returns value & 1 (bit 0 check)
  cbz r0, #skip                  ; if bit0 == 0 → stay unlocked
  movs r3, #0; str r3, [r4]      ; *out_is_unlocked = FALSE
  bl error_log(0x2014)
  movs r0, #0                     ; return AVB_IO_RESULT_OK
  pop {r4, pc}

Key: Default is_unlocked = TRUE — if callback fails/short-circuits, device treated as unlocked (verification skipped).

Note: The bit 0 check means lock_state=0 (locked, bit0=0) would be treated as unlocked. This contradiction suggests either:

  1. Lock state encoding differs from documented 0/1/2
  2. The function handles error fallthrough (unreadable seccfg = unlocked)
  3. This may be a different ops callback (not read_is_device_unlocked)

The UBFX #1 variant at 0x1a4c4 correctly isolates bit 1 (=2), matching lock_state == 2 — this is likely the real lock check, but only called from init path (0x8d2).

Self-Integrity Checks in LK

LK has multiple integrity checks that may detect and halt on patched code:

  • sha256hw_init/process/done functions at 0xA37E0, 0x137C4B
  • crc32 error! strings at 0x117664, 0x11776A
  • Image hash verification code at 0x12A640
  • ubk_hash used to verify LK's own public key root-of-trust

What likely kills a patched LK (3 hypotheses):

  1. UBK hash mismatch — LK computes hash of loaded key material, compares to fuse; patching LK changes loaded code regions, key-load function behaves differently
  2. Self-CRC32 on code section — preloader-style integrity hash embedded in LK binary; patching a single byte breaks it
  3. Xiaomi custom signature on ATF wrapper — preloader checks ATF header signature before jumping to LK; patching LK invalidates this

16. Preloader C Port Project

A buildable C reimplementation of the preloader boot flow exists at preloader_c/:

Structure

preloader_c/
Makefile, preloader.ld# Build system
gfh/header.S, gfh/gen_gfh.py # GFH headers (exact copy of original first 256 B)
include/preloader.h# All types, registers, prototypes
src/
startup.S# ARM init @ 0x100100, Thumb entry @ 0x10066C
main.c# preloader_main(): boot flow orchestrator
uart.c, printf.c, clib.c# UART 115200 8N1, minimal stdlib
wdt.c, timer.c # Watchdog, 13MHz GPT timer
dram.c# LPDDR4x skeleton
gpt.c # GPT partition parser
loader.c # Image loader, LK boot
sha256.c # Full SHA-256
jump.S# jump_to_lk(), cache flush
build/ # Build artifacts

Memory Layout (preloader_c)

Region Address Content
GFH 0x100000 GFH_FILE_INFO + 5 entries (256 B)
ARM init 0x100100 CPU setup, zero regs, BSS clear, switch to Thumb
Entry 0x10066C _preloader_entry (cache ops)
Text 0x1006A0 All C code (Thumb2)
Rodata follows String constants, K256 table
BSS follows Zero-initialized data
Stack 0x200000 SP grows down

make # → preloader_ruby_c.bin (5068 B vs original 472756 B)
