STAGE 1: BROM (Boot ROM)

Reset Vector & Early Init

On power-on, ARM CPU executes at address 0x0 in ARM mode. BROM is memory-mapped at 0x0.

Reset vector (0x00): B 0x54 — jump over exception vector table.

Exception vectors (0x04-0x3C): All infinite loops (undefined, SWI, prefetch abort, data abort — unused in BROM).

Context save (0x10-0x3C):

0x10: LDR  r9, =0x00102B60    ; Context save area base
0x14: STR  r0, [r9]           ; Save boot reason
0x18: STR  r1, [r9, #4]       ; Boot config
0x2C: MRC  p15, 0, r0, c5, c0, {0}  ; MPU region config
0x30: MRC  p15, 0, r1, c6, c0, {0}  ; MPU region base
0x3C: B    0xD9A8             ; Jump to main init

Saved context: r0 (boot device select: eMMC/UFS/SDMMC), r1-r4 (boot config), LR, CP15 c5/c6 (MPU region config for boot device window).

Main Init at 0x54

0x54: MOV  r0, #0             ; Zero all working registers (r0-r12, SP, LR)
0xB0: BL   0x11B54            ; early_hardware_init() — clocks, PLLs, watchdog
0xE0: CPS  #0x13              ; Enter SVC mode
0xE4: MOV  SP, =0x00013084    ; SVC stack pointer
0xF0: BL   0x20C              ; secure_init() — enable secure features
0xF4: BL   0x11BB0            ; clock_init() — system clock tree
0xF8: BL   0x7B20             ; watchdog_disable()
0xFC: BLX  0xDA70             ; Switch to Thumb mode → main_brom()

Main BROM (Thumb mode, 0xDA70+)

  1. Boot source detection — reads HW registers for eMMC/UFS/COMBO (strings: "EMMC_BOOT" @ 0x12B4, "UFS_BOOT", "COMBO_BOOT")
  2. Storage controller init
  3. TIMESTMP header check — validate flash layout
  4. Preloader loading — scans for GFH header (MMM\x01 magic), parses GFH_FILE_INFO, copies to SRAM at 0x100000, verifies signature via embedded public key
  5. USB Download Mode — if key combo pressed or boot fails:
      - "USBDL" mode (strings at 0x3FB0)
      - ACM USB virtual COM port modes: "ACM COMMU", "ACM DATA", "ACM VIRTUALCOM"
      - Waits for signed Download Agent (DA) via USB
  6. Jump to preloader — clear caches, BX to preloader entry at 0x10066C

BROM does NOT do: DRAM init, filesystem parsing, boot image verification.

STAGE 2: Preloader

Loading & GFH Header Chain

BROM loads preloader_ruby.bin from boot partition via raw block reads. File starts with GFH headers:

OffsetTypeSizeDescription
0x000GFH_FILE_INFO (type 0)56 BImage descriptor
0x038GFH_BL_INFO (type 1)12 BBootloader info/version
0x044GFH_BL_SEC_KEY (type 7)100 BRSA public key (data at 0x50)
0x0A8GFH_ATTEST (type 2)20 BAttestation data
0x0BCGFH_BL_LOAD (type 8)48 BLoad segment descriptors
0x0ECGFH_BL_ATTR (type 18)20 BAttributes (debug flags?)

GFH_FILE_INFO parsed:

magic:       MMM\x01
name:        FILE_INFO
flash_type:  0x0F00          ; eMMC boot partition
sig_len:     0x20 (32)       ; SHA-256 hash length
load_addr:   0x100000
entry:       0x66C           ; Entry = load_addr + 0x66C = 0x10066C

GFH_BL_SEC_KEY contains embedded RSA public key for preloader signature verification. If verification fails, BROM falls back to USB download mode.

File Layout

0x000-0x0FF:  GFH headers
0x100-0x26C:  ARM startup code
0x300-0x340:  GFH data region
0x100-0x57FFF: Thumb2 code
0x58000-0x6F000: strings / rodata
0x72000-0x73634: GOT (544 string references via LDR+ADD GOT-based pattern)

Execution Entry (0x10066C)

  • Set up stack pointer (SRAM at ~0x200000+)
  • Initialize PLLs, DRAM controller, pinmux
  • Switch to Thumb2 mode

DRAM Initialization

Primary job: init LPDDR4x via PMIC/SPI bus, calibrate timing, enable channels.

After DRAM init:

0x100000-0x17B000  Preloader code
0x200000+          Stack/heap/BSS
0x40000000+        DRAM available

LK Main (0x4827AEC) — C Runtime

  1. Architecture init: MMU + page tables, D-cache enable, TLB maintenance
  2. Platform init (MT6877-specific): UART console, clock tree, GPIO/pinmux for display/buttons
  3. Display init: Panel, backlight, framebuffer (boot logo)
  4. Boot device detection: Reads PMIC/registers for boot reason:
      - Normal boot
      - Recovery (vol up + power)
      - Fastboot (vol down + power)
      - Meta mode
  5. Boot image loading: GPT locate → read Android boot image header v3/v4 → load kernel (Image.gz-dtb) + ramdisk to DRAM → pass DTB
  6. Kernel boot: ATAGs/device tree setup → AArch64 mode → branch to kernel entry

Preloader: Two Patching Strategies

Method A — Call-site NOP (preloader_ruby_patched.bin): 12 bytes, 3 sites.

Method B — Function-return (preloader_ruby_patched_local.bin): 8 bytes, 2 sites.

Both achieve same result: no signature verification on any loaded image.

Relationship: RE_CONTEXT.md describes patches at 0x50CD4/0x50F94 (matching patched_local), but ANALYSIS_PROGRESS.md labels the "patched" file uses different sites (0x3E66C/0x3E68C/0x4618E). This suggests:

  • patched_local was first attempt (function-level bypass, more obvious, smaller footprint: 8 B)
  • patched was refined (call-site bypass, subtler: 12 B)

LK Boot Mode State Overwrite (lk_a_patched.bin)

11 bytes changed at 0x273B4-0x27400:

; Original (0x273B4):
CMP  r4, #4
IT   EQ
STREQ.W  r4, [r8]           ; Store 4 when r4==4
BEQ  0x273EA
...
; At 0x27400:
MOVS r3, #3
STR.W r3, [r8]              ; Store 3 in other path

; Patched:
MOVS r3, #0
STR.W r3, [r8]              ; Always store 0
B    0x273EA
...
; At 0x27400:
MOVS r3, #0
STR.W r3, [r8]              ; Store 0 instead of 3

Effect: Values 4 and 3 stored to [r8] are replaced with 0. [r8] likely controls boot mode lock or verified boot state (green/orange/red). Neutralizes a lock/security check in boot mode selection.

Vulnerability Surface

VULN 1 GCPU Blacklist Bypass (Hardware Crypto Engine)

Location: BROM — GCPU @ 0x10050000 CRITICAL

Type: Hardware security range enforcement bypass

BROM implements memory blacklist preventing reads from sensitive regions (BROM, preloader SRAM, security registers). Managed through GCPU registers. Blacklist registers writable through same AES-CBC interface used for crypto — no access control separation.

Exploit: Write 0 to blacklist registers via AES-CBC setup → full memory R/W.

Impact: Full BROM dump, preloader dump from SRAM, R/W to ALL memory.

VULN 2 Kamakiri — USB Control Transfer Overflow

Location: BROM — USB CDC ACM interface CRITICAL

Type: USB control transfer buffer overflow + register manipulation

Three variants:

  • v1: Write to watchdog scratch reg (0x10007050) → overwrite send_ptr → trigger SEND_CERT (cmd 0xE0) → code exec in BROM context
  • v2: USB CDC line coding ctrl_transfer — no length validation on descriptor copy → stack buffer overflow → overwrite adjacent data
  • v3 (KamakiriPL): Preloader-level USB CDC line coding overflow → bypass auth state; also bypass_2nd_da() for DA auth bypass

Impact: Arbitrary payload in BROM context, preloader/BROM dump, bypass 1st-stage download auth.

VULN 3 HeapBait — XML Parser Heap Overflow (DA)

Location: Download Agent — XML command parser CRITICAL

Type: Heap buffer overflow (allocated size vs. read size mismatch)

When DA handles CMD:SECURITY with allinone_signature:

  1. Parses XML filename attribute → allocates buffer based on ADVERTISED filename size
  2. Reads file data from USB using PACKET SIZE (0x2000), NOT allocated size
  3. Buffer allocated: 0x1400 bytes; data per chunk: 0x2000 bytes → overflow: 0x10 bytes

Exploit: 3-stage: upload shellcode as signature → trigger overflow with 5000-char filename → corrupted alloc header causes DPC callback overwrite → shellcode exec with DA privileges → patch security policies (hash binding, verify policy, download policy, SBC, SLA).

Self-test values (ARM64 on MT6886+):

shellcode_addr = 0x44C7F100, heap_base = 0x4007F100, dpc_addr = 0x40070028

Check: DA patched if contains "Dest XML file name buffer overflow".

VULN 4 Efuse/OTP Not Programmed

Location: Efuses @ 0x11F10000, OTP partition CRITICAL

Type: Missing permanent security configuration

efuse.bin: ALL 0xAA (unprogrammed erase state). otp.bin: ALL 0x00.

Missing critical efuses: secure boot enable, JTAG disable, OEM key hash, boot config. Device has no hardware root of trust — software security only.

Impact: Any software bypass is PERMANENT. Cannot be locked against after boot. All efuses R/W via GCPU (Vuln 1).

VULN 5 Security Partitions Rewritable

Location: Flash partitions seccfg, gsort, sec1, vbmeta_* HIGH

Type: Software-enforced security stored on rewritable media

  • seccfg: 64 B used out of 8 MB — magic "MMMM", flags=0x01 (enabled?), 32-byte hash/secret. Entire partition zeroable.
  • gsort: Minimal data with repeating patterns at 0x70/0x78 (rollback counter?). Erased patterns throughout.
  • sec1: Completely zero (2 MB) — no lock state = bootloader unlocked.
  • vbmeta: Standard AVB header — irrelevant if bootloader patched.

VULN 6 Preloader RSA Verify NOP (Confirmed)

Location: preloader_ruby.bin @ 0x50CD4, 0x50F94, 0x3E66C, 0x3E68C HIGH

Type: RSA signature verification bypass (two methods)

Both methods confirmed working: function-return patch (return 0) and call-site NOP (skip calls). See Patch Analysis section for byte-level details.

VULN 7 LK Boot Mode State Overwrite

Location: out/lk_a.bin @ 0x273B4, 0x27400 MEDIUM

Type: Security state register overwrite

Values 4 and 3 stored to [r8] replaced with 0 — neutralizes locked/enforced boot states. See Patch Analysis section for details.

Security Architecture

  1. BROM → Preloader: RSA-2048 + SHA-256 signature check via key in GFH_BL_SEC_KEY. Fail → USB download mode (requires authenticated DA).
  2. Preloader → LK/boot: Same RSA-2048 + SHA-256. Each image (lk, boot, recovery, tee) individually verified.
  3. LK → Kernel: Android Verified Boot (AVB) — vbmeta hash tree + rollback index. LK passes device tree with verified boot state.
  4. Lock/unlock state: Preloader checks bootloader lock state (sec1 partition). Locked = signed images only. Unlocked = warning screen + any image boots.
  5. SLA/DAA auth: Download mode requires SLA (challenge-response with OEM key) + DAA (signed DA binary).

Recommended Attack Path (Lowest Friction)

1. USB connection to device in BROM mode (vol up + power)
2. Kamakiri (Vuln 2) → BROM code execution → dump BROM, verify blacklist addresses
3. GCPU bypass (Vuln 1) → disable blacklist → full memory access
4. Dump preloader from SRAM (0x200000-0x240000)
5. Apply preloader patches (Vuln 6): NOP RSA calls
6. Write patched preloader to boot partition
7. Device boots ANY unsigned boot image

If HeapBait payload exists for MT6877:

1. Send signed DA → load HeapBait → XML overflow (Vuln 3)
2. DA security patch: hash binding, verify policy, download policy all → return 0
3. Direct flash write → replace LK, boot, recovery with unsigned versions

Cold Storage Note

Efuse state (all 0xAA) is the single most critical finding:

  • Device never had hardware secure boot enabled
  • Full software restore to stock cannot enable it
  • OEM shipped with bootloader permanently unlockable
  • Any exploit used once survives future software updates
  • OTA updates rewrite flash with locked state, but efuses unblown → software locks bypassable again via same exploits
